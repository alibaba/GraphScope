Cypher 手册概述

概述

什么是 Cypher？

Cypher 是一种声明式的图查询语言，专门为图数据库设计。它提供了一种直观且富有表现力的方式来查询、操作和管理图数据。我们的实现基于 OpenCypher  规范，这是一个图查询语言的开放标准。

与 SQL 的主要区别

虽然 SQL 是为具有表和行的关系型数据库设计的，但 Cypher 针对具有节点、关系和属性的图数据库进行了优化：

  • 结构：SQL 使用表和连接；Cypher 使用节点、关系和模式
  • 模式匹配：SQL 需要显式连接；Cypher 使用模式匹配语法
  • 遍历：SQL 需要复杂的连接来进行多跳查询；Cypher 自然支持路径遍历
  • 可读性：Cypher 的 ASCII 艺术语法使图模式在视觉上更加直观

在 NeuG 中你可以用 Cypher 做什么？

在 NeuG 中，我们将一个 Cypher 查询称为语句（Statement）。一个语句由多个**子句（Clause）**组成。例如，在以下查询中：

MATCH (p:person)
WHERE p.age = '29'
RETURN p.name as name;

其中的 MATCH、WHERE 和 RETURN 组件被称为子句，它们是图数据库操作的基本逻辑单元。

基于 OpenCypher，我们定义了一系列用于管理 NeuG 图数据库的语句语法，包括：

模式管理（DDL）

NeuG 主要面向模式严格的图数据场景，其中每一条数据都必须符合预定义的模式规范。这类似于传统的 SQL；然而，图数据涉及更复杂的节点和关系结构，这些结构也必须遵守预定义的模式要求。

例如，考虑以下模式图：

上述模式图可以通过以下语句创建：

// 示例模式定义
CREATE NODE TABLE person (
    name STRING,
    age INT32,
    PRIMARY KEY (name)
);
 
CREATE NODE TABLE software (
    name STRING,
    lang STRING,
    PRIMARY KEY (name)
);
 
CREATE REL TABLE knows (
    FROM person TO person,
    weight DOUBLE
);
 
CREATE REL TABLE created (
    FROM person TO software,
    weight DOUBLE
);

符合模式的查询： 在以下查询中，顶点标签 person 和边标签 (person-knows->person) 都符合上面定义的模式约束。person 节点包含 age 和 name 属性，且 age 属性为 INT32 类型，可以与常量 18 进行比较。因此，此查询满足所有模式约束，是有效的：

MATCH (p:person)-[:knows]->(f:person)
WHERE p.age > 18
RETURN p.name, f.name;

不符合模式的查询（会失败）： 此查询中指定的边标签 (person-follows-> person) 在模式中不存在，使其无效并导致”表 follows 不存在”错误。

MATCH (p:person)-[:follows]->(m:person)
RETURN p.name;

我们定义了一组语法来创建如上所示的模式图，我们称之为 DDL（数据定义语言）。所有后续的数据更新和查询操作都必须符合当前 DDL 定义的模式规范。我们将在 DDL 章节 中详细介绍这一点。

数据查询（DQL）

我们还定义了一套查询语法，可以同时满足事务处理（TP）和分析处理（AP）的查询需求。

例如，你可以使用以下查询在图数据库中查询所有三角形模式：

MATCH (a:person)-[:created]->(b:software),
      (c:person)-[:created]->(b:software),
      (a:person)-[:knows]->(c:person)
WHERE a.name < c.name
RETURN a.name, b.name, c.name;

我们将每个 MATCH、WHERE 和 RETURN 称为一个子句，这些是图数据操作的基本单元。在这里，MATCH 操作主要匹配构成三角形模式的所有数据，WHERE 进一步过滤模式数据以保证去重，而 RETURN 操作执行名称投影并输出最终结果。MATCH 操作主要完成图模式匹配，而 WHERE/RETURN 操作主要执行类似 SQL 的关系操作。这些子句将在 DQL 部分 中详细介绍。

为了进一步确保子句操作对数据的合法性，我们定义了 NeuG 支持的数据类型边界，以及基于这些数据类型的表达式操作。这些内容将在 数据类型 和 表达式部分 中详细介绍。

数据管理（DML）

除了 DQL 和 DDL，NeuG 还支持数据更新功能，我们称之为 DML（Data Manipulation Language）。DML 操作可以通过批量加载或增量更新来执行。

批量导入示例：

COPY person FROM "person.csv" (delim=',');
COPY knows FROM "knows.csv" (delim=',');

上述两个语句首先从 person.csv 批量加载标签为 person 的节点数据，然后从 knows.csv 批量加载标签为 person-[knows]->person 的边数据。

增量更新示例：

我们还提供了增量写入语法用于增量更新图数据。

节点创建示例：

CREATE (p:person {name: 'Bob', age: 30});

关系创建示例：

MATCH (a:person {name: 'Bob'}), (b:person {name: 'marko'})
CREATE (a)-[:knows {weight: 3.0}]->(b);

节点删除示例：

MATCH (p:person {name: 'Bob'})
DELETE p;

我们将在 DML 章节 中详细介绍这些 DML 操作。

临时加载

NeuG 提供了临时加载功能，允许用户查询外部数据源（如 CSV、JSON 和 Parquet 文件），而无需将数据导入到持久化图存储中。外部数据可以按需加载并直接查询，使此功能非常适合快速探索、转换和临时分析。

根据查询意图，NeuG 支持两种互补的加载模型：

  • LOAD FROM 将外部数据加载为临时表，支持类 SQL 操作，如投影、过滤、排序和聚合。

  • LOAD AS 将外部数据加载为临时图，支持使用 MATCH 查询进行图模式匹配和遍历。

查询示例

你可以使用 LOAD FROM 直接对外部文件执行关系操作。 例如，以下查询加载一个 CSV 文件并返回按年龄（升序）和姓名（降序）排序的前 10 条记录：

LOAD FROM "person.csv" (delim=',')
RETURN name, age
ORDER BY age ASC, name DESC
LIMIT 10;

对于更复杂的面向图的分析，外部数据可以作为临时图加载并使用图模式进行查询。

LOAD FROM "person.csv" (delim=',')
AS person;
 
LOAD FROM "knows.csv" (delim=',')
AS knows;
 
MATCH (p1:person)-[:knows*1..2]->(p2:person)
RETURN p1, p2;

这允许用户在不将图实例化到持久存储中的情况下探索多跳关系。

当前状态

目前，NeuG 完全支持通过 LOAD FROM 将外部数据加载为临时表。 你可以参考从文件加载了解详细用法和支持的操作。 通过 LOAD AS 将外部数据加载为临时图的功能目前正在开发中，详细的使用指南将在后续版本中发布。
