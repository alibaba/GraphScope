事务事务管理

事务管理

本文档描述了 NeuG 的事务管理系统，该系统根据部署模式的不同而采用不同的操作方式。NeuG 支持两种主要模式：适用于分析处理（AP）的嵌入式模式和适用于事务处理（TP）的服务模式。

设计理念

NeuG 的事务模型基于对图工作负载特征的深入理解，反映了各种权衡：

  • 图原生工作负载 主要是读取和插入操作，而更新和模式更改相对较少
  • 分析工作负载 优先考虑单查询吞吐量而非复杂的事务一致性保证
  • 事务工作负载 需要具有强隔离性的并发读/插入性能

NeuG 没有实现复杂度很高的通用事务系统，而是提供适合工作负载的保证，在简洁性、性能和正确性之间取得平衡。

事务模型

NeuG 实现了一个具有自动提交语义的隐式事务模型：

  • 无显式事务原语：NeuG 目前不提供 BEGIN、COMMIT 或 ABORT 命令
  • 自动提交语义：每个 execute() 调用都会自动形成自己的事务——每次调用执行一条语句
 
# 每个 execute() 都是一个独立的、自动提交的事务
conn.execute("CREATE (p:Person {name: 'Alice'})")  # 事务 1
conn.execute("CREATE (p:Person {name: 'Bob'})")    # 事务 2

v0.2 版本即将推出：多语句执行（在单个 execute() 调用中用 ; 分隔的多个语句）和显式事务控制（BEGIN/COMMIT/ABORT）将在 NeuG v0.2 中得到支持。

部署模式概述

方面嵌入式模式服务模式
使用场景分析、ETL、批处理交互式应用、Web 服务
并发性单用户、顺序写入多用户、并发访问
持久化显式的基于检查点的持久化自动的基于 WAL 的持久化
优化目标最大单查询性能并发读取/插入吞吐量

ACID 属性

嵌入式模式（AP 导向）

嵌入式模式专为分析工作负载设计，在这种场景中，简单性和单查询性能比复杂的事务保证更为重要。

原子性

当前限制： 对于大型写入操作，语句级别的原子性不能完全保证。

对于涉及大量数据写入的语句（例如用于批量加载的 COPY），执行中途失败可能会导致部分数据被写入内存。这是 AP 工作负载中的已知权衡，因为完整的回滚机制开销通常是不必要的。

实际解决方法： 由于在显式 CHECKPOINT 之前所有写入都保存在内存中，用户可以建立恢复点：

-- 在大型操作前创建恢复点
CHECKPOINT;
 
-- 执行批量数据加载
COPY Person FROM 'large_dataset.csv';
 
-- 如果在此点之前发生错误，重启数据库
-- 从最后一个检查点重新加载（所有未提交的更改都被丢弃）

一致性

模式约束和引用完整性在语句级别强制执行。无效操作（例如创建重复的主键或违反边约束）将被拒绝。

隔离性

所有写入操作都使用悲观独占锁。任何写入语句（插入、更新、删除或模式修改）都会获取一个独占锁，阻止所有其他操作直到完成。

操作类型并发行为
读 + 读允许并发执行
写 + 读/写相互排斥

这种简单的锁定模型对于单用户分析工作负载已经足够，并且避免了细粒度并发控制的复杂性。

持久性

更改仅在以下情况下才会持久化到磁盘：

  • 显式的 CHECKPOINT 语句，或
  • 使用 checkpoint_when_close=True（默认）关闭数据库

对于内存数据库，持久性不适用，因为数据仅存在于易失性内存中。

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
# 更改仅在内存中
conn.execute("CREATE (p:Person {name: 'Alice'})")
 
# 显式持久化到磁盘
conn.execute("CHECKPOINT")
 
# 或依赖关闭时的自动检查点
conn.close()
db.close()  # checkpoint_when_close=True 默认启用

服务模式

服务模式专为高吞吐量事务工作负载和并发客户端访问而设计。并发模型针对图原生访问模式进行了优化，其中读取和插入占主导地位。

原子性

每个语句都作为全有或全无的操作执行。失败的语句会自动回滚，其他事务不会看到任何部分状态更改。

一致性

强制执行模式约束和引用完整性。所有并发操作都会观察到数据库的一致视图。

隔离性

NeuG 使用 多版本并发控制 (MVCC) 来提供可串行化的隔离性，并具有特定操作的并发规则：

操作类型并发行为
读取与所有读取和插入操作并发
插入与读取和其他插入操作并发
更新获取全局写锁，阻塞所有操作
模式 (DDL)获取全局写锁，阻塞所有操作
检查点获取全局写锁，阻塞所有操作

设计原理： 这种混合方法反映了图工作负载的实际情况：

  • 读取和插入 是大多数图应用中的主要操作（社交网络、知识图谱、推荐系统）
  • 更新和模式更改 相对较少，可以容忍独占锁定
  • 对所有写入类型都使用完整的 MVCC 会增加显著的复杂性，而对典型的图工作负载来说收益很小
from neug import Session
 
# 并发会话
session1 = Session("http://localhost:10000/")
session2 = Session("http://localhost:10000/")
 
# 这些可以并发执行（读取 + 插入）
session1.execute("MATCH (p:Person) RETURN count(p)", access_mode="read")
session2.execute("CREATE (p:Person {name: 'Bob'})", access_mode="insert")
 
# 这将阻塞其他操作（更新需要全局锁）
session1.execute("MATCH (p:Person) SET p.updated = true", access_mode="update")

持久性

所有修改都通过**预写日志（WAL）**立即持久化：

  • 变更在执行完成前记录到 WAL
  • 崩溃恢复时自动重放 WAL 以恢复一致状态
  • 持久性不需要显式检查点（但提供用于 WAL 合并）

访问模式

执行查询时，你可以指定一个 access_mode 来控制事务行为并优化性能：

模式别名描述
readr只读操作（不带变更的 MATCH）
inserti仅插入操作（创建新顶点/边）
updateu更新/删除操作（SET、DELETE、MERGE）
schemas模式修改（创建/删除节点/边表）

默认行为： 如果未指定，NeuG 会推断适当的模式。显式指定正确的模式可以在服务模式下实现更好的并发优化。

访问模式层次： 访问模式遵循层次结构，其中较高级别的模式提供更强的保证但并发性更低：

read < insert < update ≈ schema
  • 向上兼容性：使用比所需更高级别的访问模式始终可以工作（例如，对只读查询使用 update 模式），但由于更强的锁定可能会降低吞吐量
  • 向下限制：使用比所需更低级别的访问模式会导致错误（例如，对插入操作使用 read 模式）
指定模式实际操作结果
read读取✅ OK
read插入/更新❌ 错误
insert读取/插入✅ OK
insert更新/模式❌ 错误
update/schema任意✅ OK（全局锁）
 
# 最佳实践：匹配访问模式与操作以获得最佳并发性
conn.execute("MATCH (n) RETURN n", access_mode="read")        # 仅读取锁
conn.execute("CREATE (p:Person {name: 'Alice'})", access_mode="insert")  # 插入锁
 
# 可行但非最佳：读取查询使用更新模式（获取全局锁）
conn.execute("MATCH (n) RETURN n", access_mode="update")      # 可行，但会阻塞其他操作

数据持久化

嵌入模式：基于检查点

所有更改都保存在内存中，直到显式持久化：

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
# 建立恢复点
conn.execute("CHECKPOINT")
 
# 执行操作（更改在内存中）
conn.execute("COPY Person FROM 'employees.csv'")
conn.execute("COPY Company FROM 'companies.csv'")
 
# 将所有更改持久化到磁盘
conn.execute("CHECKPOINT")
 
conn.close()
db.close()

检查点特性：

  • 创建当前数据库状态的原子快照
  • 在执行期间阻塞其他操作
  • 对于大型数据集可能需要较长时间
  • 原子性地替换之前的检查点

服务模式：基于 WAL

通过预写日志（Write-Ahead Logging）自动持久化更改：

from neug import Session
 
session = Session("http://localhost:10000/")
 
# 每个语句都会立即持久化
session.execute("CREATE (p:Person {name: 'Alice'})")  # 通过 WAL 持久化
session.execute("CREATE (p:Person {name: 'Bob'})")    # 通过 WAL 持久化
 
# 可选：将 WAL 合并到检查点以优化存储
session.execute("CHECKPOINT")
 
session.close()

服务模式检查点：

  • 暂时阻塞所有操作
  • 将 WAL 条目合并到统一的检查点中
  • 清除已处理的 WAL 条目以回收存储空间
  • 不影响单个语句的自动持久化

错误恢复

嵌入模式

恢复依赖于最后一次成功的检查点：

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
conn.execute("CHECKPOINT")  # 恢复点
 
try:
    conn.execute("COPY LargeTable FROM 'huge_file.csv'")
    conn.execute("CHECKPOINT")  # 如果成功则提交
except Exception as e:
    # 部分数据可能在内存中
    # 关闭并重新打开以从上次检查点恢复
    conn.close()
    db.close()
    
    db = neug.Database("/path/to/database")  # 从检查点恢复
    conn = db.connect()

服务模式

使用 WAL 自动崩溃恢复：

  • 未提交的操作会自动回滚
  • 数据库在启动时恢复到最后一致状态
  • 无需手动干预

最佳实践

嵌入模式

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
# 1. 创建模式并检查点
conn.execute("CREATE NODE TABLE Person(id INT64, name STRING, PRIMARY KEY(id))")
conn.execute("CREATE NODE TABLE Company(id INT64, name STRING, PRIMARY KEY(id))")
conn.execute("CREATE REL TABLE WorksAt(FROM Person TO Company)")
conn.execute("CHECKPOINT")
 
# 2. 分批加载数据并定期检查点
conn.execute("COPY Person FROM 'employees_batch1.csv'")
conn.execute("COPY Person FROM 'employees_batch2.csv'")
conn.execute("CHECKPOINT")
 
conn.execute("COPY Company FROM 'companies.csv'")
conn.execute("COPY WorksAt FROM 'employment.csv'")
conn.execute("CHECKPOINT")
 
# 3. 运行分析查询（只读，高性能）
result = conn.execute("MATCH (p:Person)-[:WorksAt]->(c:Company) RETURN c.name, count(p)")
 
conn.close()
db.close()

服务模式（TP导向）

from neug import Session
 
session = Session("http://localhost:10000/")
 
try:
    # 在不同会话间可以并发进行读取和插入操作
    session.execute("CREATE (p:Person {name: 'Alice'})", access_mode="insert")
    
    # 读取操作不会阻塞插入操作
    result = session.execute("MATCH (p:Person) RETURN count(p)", access_mode="read")
    
    # 更新操作会获取全局锁 - 在高并发场景中应谨慎使用
    session.execute("MATCH (p:Person) WHERE p.name = 'Alice' SET p.verified = true", 
                   access_mode="update")
    
    # 定期检查点以合并 WAL（可选）
    session.execute("CHECKPOINT")
    
finally:
    session.close()

总结

属性嵌入模式服务模式
原子性部分（基于检查点的恢复）完全（自动回滚）
一致性强制执行模式约束强制执行模式约束
隔离性独占写锁MVCC 用于读/插入，更新/DDL 使用独占锁
持久性显式 CHECKPOINT 或关闭自动 WAL 持久化
并发读取是是
并发插入否是
并发更新否否（全局锁）
恢复手动（检查点重载）自动（WAL 重放）

路线图

  • 多语句执行：在单个 execute() 调用中用 ; 分隔的多个语句，共享相同的事务边界
  • 显式事务原语（BEGIN/COMMIT/ABORT）以实现细粒度的事务控制
  • 保存点支持，用于事务内的部分回滚

恢复与持久性

  • 嵌入模式下更透明的恢复机制
  • 增量检查点，实现高效的增量持久化
  • 减少大数据集的检查点阻塞时间
  • 自动检查点
