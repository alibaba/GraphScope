事务管理

本文档描述 NeuG 的事务管理系统，该系统根据部署模式以不同方式运行。NeuG 支持两种主要模式：适合分析处理（AP）的 嵌入式模式 和适合事务处理（TP）的 服务模式。

设计理念

NeuG 的事务模型反映了对图工作负载特征深入理解后的权衡：

  • 图原生工作负载 以读取为主并伴有插入操作，而更新和 schema 更改相对较少
  • 分析型工作负载 优先考虑单查询吞吐量而非复杂的一致性保证
  • 事务型工作负载 需要具有强隔离性的并发读/插入性能

与其实现一个复杂的一刀切事务系统，NeuG 提供适合工作负载的保证，平衡简洁性、性能和正确性。

事务模型

NeuG 实现具有自动提交语义的 隐式事务模型：

  • 无显式事务原语：NeuG 目前不暴露 BEGIN、COMMIT 或 ABORT 命令
  • 自动提交语义：每个 execute() 调用自动形成自己的事务——每次调用一条语句
# 每个 execute() 是一个独立的、自动提交的事务
conn.execute("CREATE (p:Person {name: 'Alice'})")  # 事务 1
conn.execute("CREATE (p:Person {name: 'Bob'})")    # 事务 2

v0.2 即将推出： 多语句执行（在单个 execute() 调用中用 ; 分隔多条语句）和显式事务控制（BEGIN/COMMIT/ABORT) 将在 NeuG v0.2 中支持。

方面嵌入式模式服务模式
用例分析、ETL、批处理交互式应用、Web 服务
并发单用户、顺序写入多用户、并发访问

ACID 属性

嵌入式模式（面向 AP）

嵌入式模式专为分析型工作负载设计，简洁性和单查询性能优先于复杂的事务保证。

原子性

当前限制： 对于大型写操作，语句级原子性 未完全保证。

对于涉及大量数据写入的语句（如批量加载的 COPY），执行中途失败可能导致部分数据写入内存。这是 AP 工作负载的已知权衡，因为完整回滚机制的开销通常是不必要的。

实用解决方案： 由于所有写入都保存在内存中直到显式 CHECKPOINT，用户可以建立恢复点：

-- 在大型操作前创建恢复点
CHECKPOINT;
 
-- 执行批量数据加载
COPY Person FROM 'large_dataset.csv';
 
-- 如果在此点之前发生错误，重启数据库
-- 从上次检查点重新加载（所有未提交的更改将被丢弃）

一致性

Schema 约束和引用完整性在语句级强制执行。无效操作（如创建重复主键或违反边约束）将被拒绝。

隔离性

对所有写操作使用 悲观排他锁。任何写语句（插入、更新、删除或 schema 修改）都会获取排他锁，阻止所有其他操作直到完成。

操作类型并发行为
读 + 读允许并发执行
写 + 读/写互斥

这种简单的锁定模型足以满足单用户分析型工作负载，避免了细粒度并发控制的复杂性。

持久性

更改仅在以下情况持久化到磁盘：

  • 显式 CHECKPOINT 语句，或
  • 使用 checkpoint_when_close=True（默认）关闭数据库

对于内存数据库，持久性不适用，因为数据仅存在于易失性内存中。

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
# 更改仅存在于内存中
conn.execute("CREATE (p:Person {name: 'Alice'})")
 
# 显式持久化到磁盘
conn.execute("CHECKPOINT")
 
# 或依赖关闭时自动检查点
conn.close()
db.close()  # 默认 checkpoint_when_close=True

服务模式

服务模式专为具有并发客户端访问的高吞吐量事务型工作负载设计。并发模型针对以读取和插入为主的图原生访问模式进行了优化。

原子性

每条语句作为全有或全无的操作执行。失败的语句自动回滚，其他事务看不到部分状态更改。

一致性

强制执行 Schema 约束和引用完整性。所有并发操作都观察到数据库的一致视图。

隔离性

NeuG 使用 多版本并发控制（MVCC） 提供可序列化隔离，具有操作特定的并发规则：

操作类型并发行为
读与所有读和插入并发
插入与读和其他插入并发
更新获取全局写锁，阻塞所有操作
Schema (DDL)获取全局写锁，阻塞所有操作
检查点获取全局写锁，阻塞所有操作

设计理由： 这种混合方法反映了图工作负载的现实：

  • 读取和插入 是大多数图应用中的主导操作（社交网络、知识图谱、推荐系统）
  • 更新和 schema 更改 相对较少，可以容忍排他锁定
  • 对所有写类型使用完整 MVCC 会增加显著复杂性，而对典型图工作负载收益最小
from neug import Session
 
# 并发会话
session1 = Session("http://localhost:10000/")
session2 = Session("http://localhost:10000/")
 
# 这些可以并发执行（读 + 插入）
session1.execute("MATCH (p:Person) RETURN count(p)", access_mode="read")
session2.execute("CREATE (p:Person {name: 'Bob'})", access_mode="insert")
 
# 这将阻塞其他操作（更新获取全局锁）
session1.execute("MATCH (p:Person) SET p.updated = true", access_mode="update")

持久性

所有修改通过 预写日志（WAL） 立即持久化：

  • 更改在执行完成前记录到 WAL
  • 崩溃恢复自动重放 WAL 以恢复一致状态
  • 持久性无需显式检查点（但可用于 WAL 合并）

访问模式

执行查询时，您可以指定 access_mode 来控制事务行为并优化性能：

模式别名描述
readr只读操作（不带变更的 MATCH）
inserti仅插入操作（创建新顶点/边）
updateu更新/删除操作（SET、DELETE、MERGE）
schemasSchema 修改（创建/删除节点/边表）

默认行为： 如果未指定，NeuG 会推断适当的模式。显式指定正确的模式可在服务模式下实现更好的并发优化。

访问模式层级： 访问模式遵循层级结构，更高的模式提供更强的保证但更低的并发性：

read < insert < update ≈ schema
  • 向上兼容：使用比所需更高的访问模式始终有效（如对只读查询使用 update 模式），但由于更强的锁定可能降低吞吐量
  • 向下限制：使用比所需更低的访问模式会导致错误（如对插入操作使用 read 模式）
指定模式实际操作结果
read读✅ OK
read插入/更新❌ 错误
insert读/插入✅ OK
insert更新/schema❌ 错误
update/schema任何✅ OK（全局锁）
# 最佳：匹配访问模式与操作以获得最佳并发
conn.execute("MATCH (n) RETURN n", access_mode="read")        # 仅读锁
conn.execute("CREATE (p:Person {name: 'Alice'})", access_mode="insert")  # 插入锁
 
# 可行但次优：对读查询使用更新模式（获取全局锁）
conn.execute("MATCH (n) RETURN n", access_mode="update")      # 可行，但阻塞其他操作

数据持久化

嵌入式模式：基于检查点

所有更改保存在内存中直到显式持久化：

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
# 建立恢复点
conn.execute("CHECKPOINT")
 
# 执行操作（更改在内存中）
conn.execute("COPY Person FROM 'employees.csv'")
conn.execute("COPY Company FROM 'companies.csv'")
 
# 将所有更改持久化到磁盘
conn.execute("CHECKPOINT")
 
conn.close()
db.close()

检查点特性：

  • 创建当前数据库状态的原子快照
  • 执行期间阻塞其他操作
  • 对于大数据集可能需要大量时间
  • 原子替换先前的检查点

服务模式：基于 WAL

更改通过预写日志自动持久化：

from neug import Session
 
session = Session("http://localhost:10000/")
 
# 每条语句立即持久化
session.execute("CREATE (p:Person {name: 'Alice'})")  # 通过 WAL 持久化
session.execute("CREATE (p:Person {name: 'Bob'})")    # 通过 WAL 持久化
 
# 可选：将 WAL 合并为检查点以优化存储
session.execute("CHECKPOINT")
 
session.close()

服务模式检查点：

  • 暂时阻塞所有操作
  • 将 WAL 条目合并为统一检查点
  • 清除已处理的 WAL 条目以回收存储
  • 不影响单条语句的自动持久性

错误恢复

嵌入式模式

恢复依赖于上次成功的检查点：

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
conn.execute("CHECKPOINT")  # 恢复点
 
try:
    conn.execute("COPY LargeTable FROM 'huge_file.csv'")
    conn.execute("CHECKPOINT")  # 成功则提交
except Exception as e:
    # 部分数据可能在内存中
    # 关闭并重新打开以从上次检查点恢复
    conn.close()
    db.close()
 
    db = neug.Database("/path/to/database")  # 从检查点恢复
    conn = db.connect()

服务模式

使用 WAL 自动崩溃恢复：

  • 未提交的操作自动回滚
  • 数据库在启动时恢复到最后一致状态
  • 无需手动干预

最佳实践

嵌入式模式

import neug
 
db = neug.Database("/path/to/database")
conn = db.connect()
 
# 1. 创建 schema 和检查点
conn.execute("CREATE NODE TABLE Person(id INT64, name STRING, PRIMARY KEY(id))")
conn.execute("CREATE NODE TABLE Company(id INT64, name STRING, PRIMARY KEY(id))")
conn.execute("CREATE REL TABLE WorksAt(FROM Person TO Company)")
conn.execute("CHECKPOINT")
 
# 2. 分批加载数据并定期检查点
conn.execute("COPY Person FROM 'employees_batch1.csv'")
conn.execute("COPY Person FROM 'employees_batch2.csv'")
conn.execute("CHECKPOINT")
 
conn.execute("COPY Company FROM 'companies.csv'")
conn.execute("COPY WorksAt FROM 'employment.csv'")
conn.execute("CHECKPOINT")
 
# 3. 运行分析查询（只读、高性能）
result = conn.execute("MATCH (p:Person)-[:WorksAt]->(c:Company) RETURN c.name, count(p)")
 
conn.close()
db.close()

服务模式（面向 TP）

from neug import Session
 
session = Session("http://localhost:10000/")
 
try:
    # 读取和插入可跨会话并发
    session.execute("CREATE (p:Person {name: 'Alice'})", access_mode="insert")
 
    # 读取不阻塞插入
    result = session.execute("MATCH (p:Person) RETURN count(p)", access_mode="read")
 
    # 更新获取全局锁 - 在高并发场景中谨慎使用
    session.execute("MATCH (p:Person) WHERE p.name = 'Alice' SET p.verified = true",
                   access_mode="update")
 
    # 定期检查点合并 WAL（可选）
    session.execute("CHECKPOINT")
 
finally:
    session.close()

总结

属性嵌入式模式服务模式
原子性部分（基于检查点恢复）完整（自动回滚）
一致性强制 Schema 约束强制 Schema 约束
隔离性排他写锁读/插入使用 MVCC，更新/DDL 使用排他锁
持久性显式 CHECKPOINT 或关闭自动 WAL 持久化
并发读取是是
并发插入否是
并发更新否否（全局锁）
恢复手动（检查点重新加载）自动（WAL 重放）
